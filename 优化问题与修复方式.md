# TChat 项目优化问题与修复方式记录

> 记录时间：2026-04-11  
> 适用范围：当前仓库 `TChat`  
> 目的：把当前项目中已经识别出的主要工程问题、影响范围、建议修复方式和落地顺序沉淀为一份可执行的优化清单。

---

## 一、结论摘要

当前项目的核心问题不在“功能不够”，而在以下几个工程方向：

1. **包体积偏大，发布包需要做 ABI 精简**
2. **核心文件过大，页面与业务逻辑耦合严重**
3. **聊天并发状态设计存在潜在串配置/串请求风险**
4. **构建参数偏保守，影响开发迭代速度**
5. **数据库迁移复杂，但缺少 schema 导出与迁移测试**
6. **测试覆盖几乎为空，后续重构风险高**

建议先做能快速见效的工程优化，再做结构性重构。

---

| 优先级 | 问题 | 影响 | 建议动作 |

## 三、详细问题与修复方式

### 1. Release 不做混淆，改为优先做 ABI 精简

**问题说明**

当前确认 **不需要混淆**，因此这里不再把开启 `minifyEnabled` 作为优化方向。  
对于本项目来说，更实际的减包手段是：

- 只发布目标 ABI
- 将重量级能力按功能拆分
- 避免把调试/模拟器架构一起带进 release

**影响**

- Release 包包含多种 ABI 时体积偏大
- 用户下载不必要的原生库
- 安装包分发成本增加

**修复方式**

保留不混淆，重点调整 ABI 策略：

```kotlin
android {
    buildTypes {
        release {
            ndk {
                abiFilters.clear()
                abiFilters += "arm64-v8a"
            }
        }
    }
}
```

**补充建议**

- `debug` 继续保留 `arm64-v8a` / `armeabi-v7a` / `x86_64`，方便开发与模拟器调试
- `release` 默认只发 `arm64-v8a`
- 如果后续需要给特定渠道兼容旧设备，再单独补 `armeabi-v7a`

### 2. APK 体积偏大，原生库和资源占比明显

**问题位置**

- `app/build.gradle.kts`
- `feature-chat/src/main/assets/mermaid.min.js`
- OCR/Native 相关依赖与产物

**已观察现象**

- Debug APK 体积约 **70MB+**
- 原生库占比高
- `mermaid.min.js` 单文件约 **3MB+**
- OCR 原生库体积明显
- `junglehelper` 原生 so 较大

**问题说明**

当前 App 将多个重量级能力直接打进主 APK：

- ML Kit OCR
- Mermaid 本地资源
- JungleHelper 原生库
- 多 ABI 同时打包

**影响**

- 下载包大
- 安装慢
- 低端机磁盘与加载压力大

**修复方式**

1. **ABI 精简**
   - release 默认只发 `arm64-v8a`
   - `x86_64` 仅保留开发/模拟器
   - `armeabi-v7a` 如无明确兼容需求，可不进入正式发布包

2. **按功能拆分重型模块**
   - OCR 相关能力拆成可选功能或 flavor
   - JungleHelper 拆成独立 flavor/模块

3. **资源瘦身**
   - 评估 Mermaid 是否必须内置本地大文件
   - 检查 LaTeX/Markdown 扩展是否全部必要

4. **发布使用 App Bundle**
   - 让商店按设备下发 ABI 资源

**建议优先级**

P0，收益非常高。

### 3. `MessageSender` 使用全局 `currentConfig`，存在并发串配置风险

**问题位置**

- `data/src/main/java/com/tchat/data/MessageSender.kt`

**问题代码特征**

- `private var currentConfig: ChatConfig? = null`
- `fun setConfig(config: ChatConfig?)`

**问题说明**

`MessageSender` 是单例，并且注释中说明支持“多个聊天同时进行流式响应”。  
但当前配置是全局变量，而不是按会话维护。

这会导致在以下场景出现潜在问题：

- 用户在两个聊天间快速切换
- 单聊和群聊并行发送
- 一个页面刚设置完工具配置，另一个页面立即发消息

**影响**

- 工具配置可能串台
- `systemPrompt` 可能串会话
- `providerId/modelName/regexRules` 可能错用

**修复方式**

将配置从“全局单值”改成“按会话存储”：

```kotlin
private val sessionConfigs = ConcurrentHashMap<String, ChatConfig?>()
```

并按以下原则调整：

1. 每次发送时都显式传入不可变 `config`
2. `chatId -> config` 绑定
3. 群聊/单聊配置独立维护
4. `setConfig` 若保留，也应只作用于当前会话上下文

**建议改造目标**

- `sendMessage(chatId, content, config)`
- `generateImage(chatId, prompt, config)`
- `regenerateMessage(chatId, ..., config)`

全部不再依赖单例里的共享配置。

### 4. Provider 只保存一个 `currentCall`，并发取消语义不清晰

**问题位置**

- `network/src/main/java/com/tchat/network/provider/OpenAIProvider.kt`
- `network/src/main/java/com/tchat/network/provider/AnthropicProvider.kt`
- `network/src/main/java/com/tchat/network/provider/GeminiProvider.kt`

**问题代码特征**

- `private var currentCall: Call? = null`

**问题说明**

当前每个 Provider 内部只维护一个 `currentCall`。  
如果未来允许多个请求并发，后一次请求会覆盖前一次请求引用。

**影响**

- 取消请求可能取消错对象
- 并发流式响应管理不稳定
- “支持多会话并发”与底层实现语义不完全一致

**修复方式**

将网络请求对象改为 request-scoped：

1. 每个请求内部管理自己的 `Call`
2. 外部通过 `Job` 或 `requestId` 控制取消
3. Provider 不再保存共享 `currentCall`

可选方案：

- `Map<requestId, Call>`
- 或直接由上层协程取消 `callbackFlow`，内部自动取消对应 `Call`

### 5. `MainActivity` 职责过重

**问题位置**

- `app/src/main/java/com/tchat/wanxiaot/MainActivity.kt`

**问题说明**

`MainActivity` 当前承担了过多职责，包括但不限于：

- 初始化数据库
- 初始化 `MessageSender`
- 初始化 TTS
- 构造 Repository
- Provider 组装
- 主导航管理
- 群聊配置拼装
- 深度研究启动逻辑

该文件行数已超过千行，已经不适合继续承载新增业务。

**影响**

- 维护难
- 回归测试难
- 状态管理分散
- 页面生命周期与依赖管理耦合严重

**修复方式**

建议拆分为以下层次：

1. **AppContainer / ServiceLocator**
   - 负责数据库、Repository、Provider、TTS 等对象创建

2. **MainViewModel**
   - 负责导航状态、当前聊天/群聊/助手状态

3. **Route 层**
   - `ChatRoute`
   - `SettingsRoute`
   - `DeepResearchRoute`

4. **业务组装器**
   - 群聊配置组装
   - DeepResearch 启动器
   - Provider 创建策略

**额外建议**

如果后续继续增长，建议引入 Hilt；  
如果暂时不引入框架，至少先做一个轻量级 `AppContainer`。

### 7. Room 数据库迁移复杂，但未导出 schema

**问题位置**

- `data/src/main/java/com/tchat/data/database/AppDatabase.kt`

**问题代码特征**

- `version = 28`
- `exportSchema = false`

**问题说明**

数据库版本已经较高，迁移链条较长，并且存在较复杂的迁移逻辑。  
但当前关闭了 schema 导出，意味着：

- 无法清晰追踪每次 schema 变化
- 不利于做迁移回归测试

**影响**

- 老用户升级风险变高
- 迁移问题不容易定位
- 数据库结构演进不可追溯

**修复方式**

1. 开启 schema 导出：

```kotlin
@Database(
    ...,
    exportSchema = true
)
```

2. 在 Gradle 中配置 Room schema 输出目录
3. 将 schema 文件纳入版本控制
4. 补充 migration test，至少覆盖：
   - 关键版本到最新版
   - 复杂迁移链

**建议**

把 Migration 拆到独立文件，避免 `AppDatabase.kt` 继续膨胀。

### 8. `getChatById()` 不是响应式 Flow

**问题位置**

- `data/src/main/java/com/tchat/data/database/dao/ChatDao.kt`
- `data/src/main/java/com/tchat/data/repository/impl/ChatRepositoryImpl.kt`

**问题说明**

当前 `ChatRepositoryImpl.getChatById()` 是：

- 先调用一次 suspend DAO
- 再用 `flow { emit(...) }` 包装

这不是 Room 的响应式监听，而是“一次查询伪装成 Flow”。

**影响**

- Chat 标题/时间更新后，依赖此 Flow 的地方不会自动刷新
- 状态与 Compose 响应式模型不匹配

**修复方式**

DAO 改为直接返回 Flow：

```kotlin
@Query("SELECT * FROM chats WHERE id = :chatId")
fun observeChatById(chatId: String): Flow<ChatEntity?>
```

Repository 层直接 `map { it?.toChat() }`。

### 9. Gradle 构建参数过于保守

**问题位置**

- `gradle.properties`

**当前现状**

- `org.gradle.workers.max=1`
- `org.gradle.vfs.watch=false`
- `org.gradle.parallel` 未开启
- `org.gradle.jvmargs=-Xmx2048m`

**问题说明**

对于当前这种多模块 + Compose + NDK 项目，这套参数较保守，  
会直接影响增量构建和日常开发体验。

**影响**

- 编译慢
- assemble/install 速度慢
- UI 修改反馈慢

**修复方式**

建议逐步调整：

```properties
org.gradle.jvmargs=-Xmx4096m -Dfile.encoding=UTF-8
org.gradle.parallel=true
org.gradle.vfs.watch=true
```

并移除：

```properties
org.gradle.workers.max=1
```

**注意**

如果最初是为了绕过 Windows 文件锁问题而设置这些参数，建议逐项恢复并验证，而不是一次全部打开。

### 10. 网络日志保存完整请求与响应，存在内存和隐私风险

**问题位置**

- `network/src/main/java/com/tchat/network/log/NetworkLogger.kt`

**问题说明**

日志系统会把完整：

- `requestBody`
- `responseBody`

保存在内存列表中，并保留最多 100 条。

AI 场景下这类内容往往较大，且可能包含敏感信息。

**影响**

- 内存占用增加
- 调试页面卡顿风险
- 存在隐私泄露风险

**修复方式**

1. 仅在 `debug` 构建启用完整日志
2. `release` 只记录摘要信息
3. 对 body 做长度截断，例如只保留前 4KB / 8KB
4. 更大日志落本地文件，不常驻内存
5. API Key、Token 等敏感字段继续脱敏

### 11. TTS 存在两套实现路径

**问题位置**

- `app/src/main/java/com/tchat/wanxiaot/MainActivity.kt`
- `data/src/main/java/com/tchat/data/MessageSender.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/ChatScreen.kt`

**问题说明**

项目中已经有统一的 `TtsService`，但 `ChatScreen` 内部仍然直接创建 `TextToSpeech` 实例。

**影响**

- 生命周期重复管理
- 配置可能不一致
- 页面级和全局级 TTS 行为容易分叉

**修复方式**

统一走 `TtsService`：

- 自动朗读
- 手动点击朗读
- 设置页引擎切换

都收敛到同一套服务实现中。

### 13. 序列化方案不统一

**问题位置**

- `SettingsManager.kt`
- `ExportDataModels.kt`
- `ExportImportManager.kt`
- `KnowledgeService.kt`
- `DeepResearchHistoryRepository.kt`

**问题说明**

当前项目混用了多种序列化方式：

- 手写 `JSONObject/JSONArray`
- Gson
- 多处重复 JSON 组装与解析

**影响**

- 可维护性差
- 导出/导入/DB 存储格式容易分叉
- 重构成本高

**修复方式**

建议统一到单一方案：

- 优先考虑 `kotlinx.serialization`
- DTO / 导入导出 / 配置持久化统一走同一套模型
- Room 中复杂字段使用 `TypeConverter`

**顺手可优化**

像 `floatArrayToJson/jsonToFloatArray` 这类重复逻辑应统一下沉到共享工具层。

### 14. 依赖版本管理还可以进一步统一

**问题位置**

- `gradle/libs.versions.toml`
- `data/build.gradle.kts`
- `feature-chat/build.gradle.kts`

**问题说明**

当前已有 version catalog，但部分依赖仍直接写死在模块里。

例如：

- `data` 模块手写了 `okhttp` / `okhttp-sse`
- `Gson` 等版本也散落在模块配置中

**影响**

- 升级依赖时容易遗漏
- 同类依赖可能跨版本并存

**修复方式**

1. 尽量全部迁移到 `libs.versions.toml`
2. 同一技术栈依赖统一版本来源
3. 减少“模块私有硬编码版本”

### 15. 存在疑似未使用/过时代码

**问题位置**

- `network/src/main/java/com/tchat/network/api/OpenAIService.kt`

**问题说明**

该类目前看更像旧实现，未见明显使用路径。

**影响**

- 增加认知负担
- 容易误导后续维护
- 重构时增加噪音

2. 若未使用，则删除
3. 若保留做兼容参考，移到 `legacy/` 或明确标注废弃

## 四、建议修复顺序

### 第一阶段：先做收益最高的工程优化

1. Release 默认只发 `arm64-v8a`

3. 调整 Gradle 构建参数
4. 网络日志改为 debug-only + body 截断

7. Provider 改为 request-scoped call
8. `ChatDao.getChatById()` 改为响应式 Flow
9. 统一 TTS 调用路径

### 第三阶段：做结构性重构

10. 拆 `MainActivity`
11. 拆 `SettingsScreen` / `ProviderEditScreen` / `ExportImportScreenEnhanced`
12. 抽象 `ChatViewModel` / `GroupChatViewModel` 公共逻辑
13. 统一序列化方案

### 第四阶段：补齐质量保障

14. 打开 Room schema 导出
