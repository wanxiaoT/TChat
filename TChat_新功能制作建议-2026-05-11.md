## 一、当前源码中的功能地基

本次源码阅读显示，TChat Android 已经不是单纯聊天壳，而是具备以下能力：

- 多服务商与模型能力体系：`ProviderDefinition` / `ProviderCapability` / `ModelCapability` 已经能描述聊天、流式、工具、视觉、视频输入、图片生成、模型列表等能力。
- 普通聊天与群聊：`ChatViewModel`、`GroupChatViewModel`、`GroupChatRepositoryImpl` 已支持单聊、群聊、不同助手配置、手动/轮流/随机/自然发言策略的基础逻辑。
- 消息结构较完整：`MessagePart` 已支持文本、图片、视频、工具调用、工具结果；`MessageVariant` 已支持重新生成后的多版本回复。
- 知识库 RAG：`KnowledgeService` 已支持文本、URL、文件加载，分块、Embedding、向量检索。
- 深度研究：`DeepResearchService` 已支持搜索查询生成、搜索结果处理、递归研究、最终 Markdown 报告生成、历史记录。
- 工具系统：本地工具包括文件系统、网页抓取、系统信息、延迟回复；MCP 工具也能转换为统一 `Tool`。
- Skills 系统：`Skill`、`SkillService`、`SkillMatcher` 已支持技能触发、提示注入和技能工具。
- 导入导出与备份：`ExportImportManager`、R2 备份、二维码导入导出、加密导出已有基础。
- TTS、OCR、JungleHelper、日志查看、网络日志等附加能力已经存在。

这些能力的共同问题是：许多能力是“分散可用”，但还没有被组织成高价值工作流。下面的建议优先选择可以复用现有结构、少碰后端、用户感知强的功能。

## 二、优先推荐功能

### P0-1. 全局聊天内容搜索

**建议结论**

优先做。当前侧边栏搜索主要过滤聊天标题和群聊名称，用户一旦聊天记录变多，很难按消息内容、工具结果、图片文件名、模型名、助手名找回历史。

**源码依据**

- 侧边栏已有搜索框，但过滤条件只看 `chat.title` / `groupChat.name`：`app/src/main/java/com/tchat/wanxiaot/ui/DrawerContent.kt`
- 消息 DAO 已有按聊天查询、分页历史能力，但没有面向全局内容搜索的接口：`data/src/main/java/com/tchat/data/database/dao/MessageDao.kt`
- `MessagePart` 已承载文本、工具结果、图片/视频元信息：`data/src/main/java/com/tchat/data/model/Message.kt`

**用户价值**

- 找回某次模型回答、某段代码、某个工具调用结果。
- 支持“我之前问过 xxx 在哪里”的真实使用场景。
- 对本地开源客户端尤其重要，不依赖后端。

**建议实现**

- 第一阶段：Room `LIKE` 搜索 `messages.content` / `partsJson` / `toolResultsJson`，支持关键词、聊天标题、助手名、模型名过滤。
- 第二阶段：引入 Room FTS5 虚表，索引文本消息、工具结果和知识库条目标题。
- UI 放在侧边栏搜索升级版或单独“搜索”页面。
- 点击搜索结果跳转到对应 chat，并滚动定位到消息。

**涉及文件**

- `data/src/main/java/com/tchat/data/database/dao/MessageDao.kt`
- `data/src/main/java/com/tchat/data/database/AppDatabase.kt`
- `data/src/main/java/com/tchat/data/repository/ChatRepository.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/MessageList.kt`
- `app/src/main/java/com/tchat/wanxiaot/ui/DrawerContent.kt`

**验证建议**

- 搜索普通文本消息。
- 搜索工具执行结果。
- 搜索多变体消息当前版本与历史版本。
- 删除聊天后搜索索引同步移除。

### P0-2. 深度研究报告一键入库 / 发送到聊天

**建议结论**

优先做。深度研究已经能生成报告和来源列表，知识库也已经能处理文本条目，两者天然可以连接。

**源码依据**

- 深度研究最终报告是 Markdown：`data/src/main/java/com/tchat/data/deepresearch/service/DeepResearchService.kt`
- 深度研究有历史记录：`data/src/main/java/com/tchat/data/database/dao/DeepResearchHistoryDao.kt`
- 知识库支持 `addNote()` 添加文本笔记：`app/src/main/java/com/tchat/wanxiaot/ui/knowledge/KnowledgeViewModel.kt`
- 聊天页面可从输入框启动深度研究：`feature-chat/src/main/java/com/tchat/feature/chat/ChatScreen.kt`

**用户价值**

- 深度研究不只是一次性报告，而是可沉淀为可检索知识。
- 用户可以把报告直接发送回当前聊天，继续追问、改写、总结。
- 与现有 RAG 能力组合后，TChat 会更像个人研究助手。

**建议实现**

- 在 `ResearchResult` 页面增加操作：
  - 保存到知识库
  - 发送到当前聊天
  - 导出 Markdown
  - 复制带来源报告
- 保存到知识库时，生成一个 `KnowledgeItemEntity`，类型为 `TEXT`，标题用研究 query，内容为报告正文 + 来源。
- 如果报告从聊天页面启动，保留启动来源 `chatId`，完成后支持“插入到原聊天”。

**涉及文件**

- `app/src/main/java/com/tchat/wanxiaot/ui/deepresearch/DeepResearchScreen.kt`
- `app/src/main/java/com/tchat/wanxiaot/ui/deepresearch/DeepResearchViewModel.kt`
- `app/src/main/java/com/tchat/wanxiaot/ui/knowledge/KnowledgeViewModel.kt`
- `data/src/main/java/com/tchat/data/deepresearch/repository/DeepResearchHistoryRepository.kt`
- `data/src/main/java/com/tchat/data/service/KnowledgeService.kt`

**验证建议**

- 研究完成后保存到知识库，处理状态变为待处理。
- 对保存后的研究报告执行知识库检索。
- 从聊天启动研究，研究完成后能回填到原聊天。

### P0-3. 群聊自动模式补全与群聊统计

**建议结论**

优先做。代码里已经有 `autoModeEnabled`、`autoModeDelay` 和 `GroupChatStats`，但自动模式逻辑和统计仍是 TODO，属于已经建模但没收口的功能。

**源码依据**

- `GroupChat` 已包含 `autoModeEnabled` 和 `autoModeDelay`：`data/src/main/java/com/tchat/data/model/GroupChat.kt`
- `GroupChatViewModel` 里有 `TODO: 实现自动模式逻辑`：`feature-chat/src/main/java/com/tchat/feature/chat/GroupChatViewModel.kt`
- `GroupChatRepositoryImpl.getGroupStats()` 仍返回空统计：`data/src/main/java/com/tchat/data/repository/impl/GroupChatRepositoryImpl.kt`
- 群聊消息元数据已有 `assistantId` / `assistantName`：`data/src/main/java/com/tchat/data/model/GroupChat.kt`

**用户价值**

- 多助手协作的核心亮点会更完整。
- 自动连续讨论适合头脑风暴、方案评审、模拟辩论。
- 群聊统计能帮助用户判断哪个助手贡献更多、谁经常沉默。

**建议实现**

- 自动模式：
  - 用户发消息后，根据策略选择第一个助手。
  - 每次助手回复完成后，等待 `autoModeDelay` 秒。
  - 根据 `generationMode` 和发言策略选择下一个助手。
  - 提供“暂停自动讨论”和“继续自动讨论”。
  - 设置最大轮数，避免无限循环。
- 统计：
  - 从 `messages.groupMetadata` 统计每个助手发言次数、最近发言时间、平均输出 tokens。
  - 群聊详情页显示成员贡献分布。

**涉及文件**

- `feature-chat/src/main/java/com/tchat/feature/chat/GroupChatViewModel.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/GroupChatScreen.kt`
- `data/src/main/java/com/tchat/data/repository/impl/GroupChatRepositoryImpl.kt`
- `data/src/main/java/com/tchat/data/database/dao/MessageDao.kt`
- `app/src/main/java/com/tchat/wanxiaot/ui/groupchat/GroupChatScreen.kt`

**验证建议**

- 手动模式不被自动模式影响。
- 轮流模式能按优先级轮流。
- 自动模式可取消，取消后不会继续发起网络请求。
- 群聊统计能正确识别历史消息里的 assistantId。

## 三、中优先级功能

### P1-1. 知识库处理队列、进度、取消与重试

**建议结论**

建议做。知识库现在有 `PENDING/PROCESSING/COMPLETED/FAILED` 状态，但处理体验偏粗：`processAllPending()` 顺序处理并返回结果列表，缺少进度、取消、失败重试批处理界面。

**源码依据**

- `KnowledgeItemEntity` 已有处理状态。
- `KnowledgeService.processItem()` 和 `processAllPending()` 已有基础处理流程。
- Embedding Provider cancellation 本轮已修复，为知识库取消能力打好基础。

**建议实现**

- 引入 `KnowledgeProcessingJob` 状态流：
  - 总数、已完成、失败、当前条目、当前 chunk。
  - 支持取消当前知识库处理。
  - 支持只重试失败项。
- UI 在知识库详情页显示处理队列。
- 失败项保留错误信息并可单独重试。

**价值**

- 大文件或多个 URL 导入时可控，不会让用户觉得卡死。
- 与本轮 embedding active calls 修复形成闭环。

### P1-2. 提示词 / 快捷消息库

**建议结论**

建议做。`QuickMessage` 数据模型已经存在，但当前没有明显完整 UI 和持久化工作流。提示词库是低风险、高频功能。

**源码依据**

- `QuickMessage` 在 `data/src/main/java/com/tchat/data/model/Assistant.kt` 中已定义。
- Skills、助手系统、聊天输入框都可以复用提示词片段。

**建议实现**

- 新增“快捷消息 / 提示词库”：
  - 全局提示词
  - 助手专属提示词
  - 聊天输入框 `/` 或按钮插入
  - 支持变量占位符，例如 `{topic}`、`{language}`、`{format}`
- 后续可以和 Skills 合并：普通用户用快捷消息，高级用户用 Skills。

**价值**

- 用户不用每次重复输入“翻译成中文、总结成表格、提取行动项”。
- 很适合移动端输入效率低的环境。

### P1-3. 工具调用安全中心

**建议结论**

建议做。TChat 已支持本地文件系统、网页抓取、系统信息、MCP 工具。工具能力越强，越需要可解释、可撤销、可限制。

**源码依据**

- 本地工具在 `data/src/main/java/com/tchat/data/tool/LocalTools.kt`。
- MCP 工具在 `data/src/main/java/com/tchat/data/mcp/McpToolService.kt`。
- 消息结构已能显示工具调用和工具结果：`MessagePart.ToolCall` / `MessagePart.ToolResult`。

**建议实现**

- 每个助手增加工具安全策略：
  - 自动执行
  - 每次确认
  - 只读模式
  - 禁止危险路径
- 文件系统工具增加根目录白名单。
- MCP 工具首次执行前展示 server、tool、arguments。
- 工具调用历史可从消息里展开，也可在设置页汇总。

**价值**

- 降低用户误操作风险。
- 对开源客户端更重要，因为用户可能配置本地 MCP 或文件系统工具。

### P1-4. 消息分支 / 变体对比

**建议结论**

建议做。当前 `MessageVariant` 已支持重新生成后的多版本，但 UI 只是切换版本。下一步可以把它升级成“分支对话”和“版本对比”。

**源码依据**

- `MessageVariant` 已记录内容、tokens、速度、首 token 延迟。
- `ChatRepositoryImpl.regenerateMessage()` 已能追加变体。
- `MessageItem` 已有变体选择 UI。

**建议实现**

- 对同一条 AI 回复：
  - 显示多个变体的对比视图。
  - 支持“从此版本继续新会话”。
  - 支持“设为最佳并删除其他版本”。
  - 显示每个版本的模型、tokens、速度、成本估算。

**价值**

- 让重新生成不只是覆盖，而是可比较、可复盘。
- 适合调参、对比不同模型输出。

## 四、可作为后续阶段的功能

### P2-1. 聊天自动摘要与长期记忆

当前已有 `contextMessageSize` 控制上下文长度，也有知识库能力。可以增加：

- 长聊天自动摘要。
- 对话结束后提取“用户偏好 / 项目事实 / 待办事项”。
- 用户确认后写入个人知识库。
- 发送请求时自动注入相关记忆。

需要注意隐私：默认关闭，写入前必须让用户确认。

### P2-2. 图片生成工作台

当前聊天输入框已有“生成图片”入口，Provider 接口也有 `generateImage()`。可以继续做：

- 图片尺寸、质量、数量设置。
- 图片历史画廊。
- 保存到系统相册。
- 复用某张图片继续追问或变体生成。
- 记录生成参数和模型。

### P2-3. 本地模型启动向导

`BuiltinProviderDefinitions` 已包含 `Ollama / LM Studio`，但普通用户不一定知道如何填 endpoint 和模型。可以新增：

- 自动测试 `http://127.0.0.1:11434/v1`。
- 拉取本地模型列表。
- 一键创建本地服务商配置。
- 显示“手机访问电脑本地模型”时需要局域网 IP 的提示。

### P2-4. 技能市场 / 技能包导入导出

已有 Skills 系统和导入导出框架，可以做：

- 内置技能分类。
- 技能包二维码分享。
- 技能导入时预览权限和工具。
- 助手详情页一键启用推荐技能组合。

注意不要引入远程市场后端依赖，第一阶段先做本地文件/二维码分享即可。

## 五、不建议现在优先做的功能

- 大规模后端账号体系：会越过 Android 开源客户端边界，且容易碰支付、授权、密钥管理，不适合放进本仓库。
- 复杂多人实时同步：需要服务器状态和冲突解决，不适合作为当前 Android 单机客户端的下一步。
- 全量插件市场：当前 Skills/MCP 还需要安全中心和导入导出稳定性，先做本地能力闭环更稳。
- 大规模 UI 重写：当前更需要把现有功能串成闭环，而不是另起一套设计。

## 六、推荐落地顺序

### 第一批：最快产生用户价值

1. 全局聊天内容搜索。
2. 深度研究报告一键保存到知识库 / 发送到聊天。
3. 群聊自动模式补全与群聊统计。

### 第二批：提升稳定性和可控性

1. 知识库处理队列、进度、取消、重试。
2. 工具调用安全中心。
3. 提示词 / 快捷消息库。

### 第三批：增强高级用户体验

1. 消息分支 / 变体对比。
2. 图片生成工作台。
3. 本地模型启动向导。
4. 技能包导入导出。

## 七、建议下一个实际开发任务

如果只选一个功能开始，建议做：

**全局聊天内容搜索**

原因：

- 不依赖后端，不碰支付、不碰密钥。
- 数据都在本地 Room，技术闭环清晰。
- 对所有用户都有价值。
- 可以先用简单 SQL 做 MVP，再升级 FTS。
- 能顺手补测试：消息插入、搜索命中、聊天删除后索引清理。

建议 MVP 验收标准：

- 设置或侧边栏里有全局搜索入口。
- 输入关键词能搜到消息正文。
- 搜索结果显示聊天标题、消息片段、时间、角色。
- 点击结果能进入对应聊天。
- 支持搜索群聊消息。
- 删除聊天后结果不再出现。
