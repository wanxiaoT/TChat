# TChat Android 性能优化建议

> 日期：2026-05-11  
> 范围：仅检查 Android 开源客户端 `C:\Users\Administrator\AndroidStudioProjects\TChat`。  
> 说明：本文基于代码静态检查整理，未包含真机 Profiler、Macrobenchmark 或线上埋点数据。

## 结论

当前项目已经有一些正确的性能设计：

- 聊天消息列表不是一次性加载全量历史，而是使用最近消息 + 向上翻页。
- `messages` 表已有 `chatId`、`chatId + timestamp`、收藏、模型统计等常用索引。
- 图片预览使用采样解码，并有 24MB 内存 LRU 缓存。
- 网络日志在 Release 默认关闭，并限制单条 body 捕获长度。

仍建议优先优化以下几类问题：

1. 流式输出期间的 Markdown/Markwon 解析和 Compose 重组。
2. 知识库向量检索的全量线性扫描。
3. 全局消息搜索对 `partsJson` 的 `LIKE '%query%'` 扫描。
4. 多模态上下文重复 Base64 编码大文件。
5. Release 未启用 R8、资源压缩和 Baseline Profile。

## 优先级总览

| 优先级 | 优化项 | 主要收益 | 风险 |
| --- | --- | --- | --- |
| P0 | 流式输出期间降低 Markdown 渲染频率 | 改善长回答卡顿、掉帧、输入延迟 | 中 |
| P0 | 消息搜索改为 FTS5 或搜索索引表 | 大幅降低聊天记录搜索耗时 | 中 |
| P0 | 知识库检索避免全量 JSON 向量扫描 | 大幅降低大知识库检索耗时和 GC | 中高 |
| P1 | 限制历史媒体 Base64 编码 | 降低大图/视频发送内存峰值 | 中 |
| P1 | Release 开启 R8/shrinkResources | 降低包体、方法数和冷启动压力 | 中 |
| P1 | Room 实体到 UI 模型的 JSON 解析缓存/后台化 | 降低列表刷新和数据库 Flow 更新开销 | 中 |
| P2 | Mermaid WebView 延迟渲染/缓存 | 降低含 Mermaid 消息的内存和滚动压力 | 中 |
| P2 | 拆分 Compose 状态粒度 | 降低全局设置变化导致的大范围重组 | 中 |
| P2 | 复用 OkHttpClient | 减少线程、连接池和 TLS 握手开销 | 低中 |
| P3 | 网络日志快照优化 | 改善 Debug 日志页刷新性能 | 低 |

## P0：流式输出与 Markdown 渲染

### 现状

相关文件：

- `app/src/main/java/com/tchat/wanxiaot/MainActivity.kt`
- `data/src/main/java/com/tchat/data/MessageSender.kt`
- `data/src/main/java/com/tchat/data/repository/impl/ChatRepositoryImpl.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/markdown/MarkdownText.kt`

当前 `MessageSender` 使用 `CoroutineScope(SupervisorJob() + Dispatchers.Main)`。发送任务默认从 Main dispatcher 启动，仓库层每 `48ms` 触发一次流式消息更新：

```kotlin
const val STREAMING_UPDATE_INTERVAL_MS = 48L
```

UI 侧 `MarkdownText` 在内容变化时会执行：

- `ContentParser.parse(markdown)`
- `markwon.toMarkdown(content)`
- `markwon.setParsedMarkdown(textView, spanned)`

长回答、代码块、表格、LaTeX、Mermaid 或工具调用结果较长时，流式更新会不断触发解析和 AndroidView 更新，容易造成主线程帧耗时超标。

### 建议

1. 流式期间先使用轻量纯文本渲染。
   - `message.isStreaming == true` 时不走完整 Markwon。
   - 消息完成后再进行 Markdown、LaTeX、表格、Mermaid 渲染。

2. 调整流式 UI 更新节流。
   - 将 `48ms` 调整为 `80ms` 到 `120ms`。
   - 或基于字符增量、代码块状态和是否接近列表底部动态节流。

3. 后台化文本处理。
   - 正则流处理、Markdown 分块、消息 JSON 转换放到 `Dispatchers.Default` 或 `Dispatchers.IO`。
   - Main dispatcher 只负责提交最终 UI state。

4. 给 Markdown 解析加缓存。
   - 缓存 key 可使用 `messageId + contentHash + themeKey + selectable`。
   - 对历史消息尤其有效，避免滚动回可见区域时重复解析。

5. `LazyColumn` 增加 `contentType`。
   - 用户消息、助手消息、流式消息、历史加载项可以使用不同 `contentType`。
   - 有助于 Compose 复用列表项结构。

### 验证

- 使用 5,000 字以上 Markdown 回答测试流式输出。
- 包含代码块、表格、LaTeX、工具结果和 Mermaid 的回答分别测试。
- 使用 Android Studio Profiler 观察 Main thread、GC 和分配热点。
- 使用：

```powershell
adb shell dumpsys gfxinfo com.tchat.wanxiaot framestats
```

重点观察：

- Janky frames 数量。
- 90/95/99 分位帧耗时。
- 流式输出期间输入框是否有明显延迟。

## P0：消息搜索改为 FTS5 或搜索索引表

### 现状

相关文件：

- `data/src/main/java/com/tchat/data/database/dao/MessageDao.kt`
- `data/src/main/java/com/tchat/data/repository/ChatSearchRepository.kt`
- `app/src/main/java/com/tchat/wanxiaot/ui/DrawerContent.kt`

当前全局消息搜索使用：

```sql
LOWER(messages.partsJson) LIKE LOWER(:pattern)
```

并且 pattern 是 `%query%`。这种写法通常无法使用普通索引，会随着消息数量增长退化为全表扫描。命中后还会反序列化 `partsJson` 和 `variantsJson` 生成 snippet。

### 建议方案 A：Room FTS5

新增 FTS 表保存可搜索文本：

- `messageId`
- `chatId`
- `chatTitle`
- `role`
- `contentText`
- `modelName`
- `providerId`
- `groupAssistantName`

消息新增、更新、删除时同步更新 FTS 表。搜索时使用 `MATCH`，不要扫 `partsJson`。

适合：

- 消息量大。
- 搜索体验是核心功能。
- 需要支持分词、排序、snippet。

### 建议方案 B：普通搜索索引表

如果暂时不引入 FTS，可新增普通表：

- `messageId`
- `chatId`
- `normalizedText`
- `updatedAt`

至少把 JSON 解析从搜索时挪到写入时，搜索时只查纯文本列。

适合：

- 先快速降低 JSON 解析和字段扫描成本。
- 后续再迁移 FTS。

### 验证

准备以下数据规模：

- 1,000 条消息。
- 10,000 条消息。
- 50,000 条消息。

记录：

- 首次搜索耗时。
- 连续输入搜索耗时。
- 抽屉 UI 是否掉帧。
- Room query explain plan 是否命中索引或 FTS。

## P0：知识库向量检索优化

### 现状

相关文件：

- `data/src/main/java/com/tchat/data/service/KnowledgeService.kt`
- `data/src/main/java/com/tchat/data/database/dao/KnowledgeChunkDao.kt`
- `data/src/main/java/com/tchat/data/database/entity/KnowledgeChunkEntity.kt`

当前检索流程：

1. 查询某知识库全部 chunk。
2. 每个 chunk 的 embedding 从 JSON 字符串解析成 `FloatArray`。
3. 逐个计算 cosine similarity。
4. 全量 `sortedByDescending`。
5. `take(topK)`。
6. 命中 threshold 时逐条查询 item。

这一流程在小知识库可接受，但在几千到几万 chunk 后会出现明显问题：

- JSON 解析 CPU 和内存分配高。
- 全量排序成本高。
- 逐条 `getItemById` 造成 N+1 查询。
- 每次检索都重复解析同一批 embedding。

### 建议

1. embedding 存储从 JSON 字符串改为 BLOB。
   - 使用 Float32 little-endian。
   - 或存归一化后的向量，检索时只做 dot product。

2. 使用 topK 小根堆代替全量排序。
   - 遍历时只维护 K 个最佳结果。
   - 避免对全部候选排序。

3. 先筛 chunk，再批量查 item。
   - 先得到 topK chunk。
   - 用 `WHERE id IN (...)` 批量查询 item。

4. 增加候选集预过滤。
   - 对 chunk content 建 FTS。
   - 先用关键词召回候选，再做向量相似度。

5. 中长期考虑近似向量索引。
   - 移动端可评估 HNSW、SQLite 扩展或服务端检索。
   - 若保持纯客户端实现，至少需要缓存已解析向量。

### 验证

准备知识库：

- 1,000 chunks。
- 10,000 chunks。
- 50,000 chunks。

记录：

- 查询向量耗时。
- 本地相似度计算耗时。
- JSON/BLOB 解析耗时。
- GC 次数。
- 单次检索总耗时。

## P1：限制历史媒体 Base64 编码

### 现状

相关文件：

- `data/src/main/java/com/tchat/data/repository/impl/ChatRepositoryImpl.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/ChatMediaUtils.kt`

`buildUserContentParts` 会对消息中的图片、视频文件执行 Base64 编码：

```kotlin
Base64.getEncoder().wrap(output).use { encodedOutput ->
    input.copyTo(encodedOutput)
}
return output.toString(Charsets.UTF_8.name())
```

Base64 会让数据膨胀约 33%，再转成 String 会进一步增加内存压力。若上下文消息中包含历史媒体，每次发送都可能重复编码。

### 建议

1. 默认只发送当前用户消息的媒体。
   - 历史媒体转成文本占位，例如“历史图片已省略”。

2. 图片发送前压缩。
   - 限制最长边。
   - 限制 JPEG/WebP 质量。
   - 限制最终字节数。

3. 视频设置严格上限。
   - 超过阈值直接拒绝或提示压缩。
   - 非 Gemini 模型继续使用文字占位。

4. 如果 provider 支持文件上传，优先使用文件引用。
   - 避免每次请求体内嵌大 Base64。

### 验证

- 发送 5MB、20MB、50MB 图片或视频。
- 观察内存峰值和 OOM 风险。
- 观察请求构建耗时。
- 测试历史上下文带媒体时的重复发送行为。

## P1：Release 开启 R8、资源压缩和 Baseline Profile

### 现状

相关文件：

- `app/build.gradle.kts`

当前 Release 配置：

```kotlin
release {
    isMinifyEnabled = false
    isShrinkResources = false
}
```

这会影响：

- APK 体积。
- 冷启动类加载。
- 方法数和 dex 体积。
- 未使用资源打包。

### 建议

1. 开启 R8 和资源压缩。

```kotlin
release {
    isMinifyEnabled = true
    isShrinkResources = true
    proguardFiles(
        getDefaultProguardFile("proguard-android-optimize.txt"),
        "proguard-rules.pro"
    )
}
```

2. 补充 keep 规则。
   - Room schema/entity/DAO 按需验证。
   - Markwon、JLatexMath。
   - ML Kit。
   - ImGui/native 相关入口。
   - 通过反射访问的模型类。

3. 加 Baseline Profile。
   - 覆盖冷启动、进入聊天页、打开抽屉、发送消息、打开设置页。

### 验证

- `./gradlew :app:assembleRelease`
- 安装 release 包真机测试核心路径。
- 比较 APK 大小、冷启动耗时和首次进入聊天页耗时。

## P1：Room 转换与 JSON 解析优化

### 现状

相关文件：

- `data/src/main/java/com/tchat/data/repository/impl/ChatRepositoryImpl.kt`
- `data/src/main/java/com/tchat/data/util/MessagePartSerializer.kt`

`observeRecentMessages` 每次 Room Flow 发射时会把实体列表整体转换为 `Message`：

```kotlin
entities.map { it.toMessage() }
```

`toMessage()` 内部会解析：

- `partsJson`
- `variantsJson`
- group metadata

最近 100 条消息中如果有长工具结果或多个变体，解析成本会比较高。

### 建议

1. 转换放到后台 dispatcher。
   - 数据库 Flow mapping 使用 `flowOn(Dispatchers.Default)` 或在 repository 内部明确后台转换。

2. 增加解析缓存。
   - key：`messageId + partsJson.hashCode() + variantsJson.hashCode()`。
   - 对滚动返回历史消息、收藏搜索结果等场景有效。

3. 引入轻量列表模型。
   - 列表只需要 role、文本摘要、媒体数量、是否收藏等字段。
   - 展开详情、复制、重新生成时再解析完整 parts。

4. 中长期拆表。
   - `messages` 保存核心字段和纯文本。
   - 媒体、工具调用、工具结果、变体拆到独立表。

### 验证

- 构造 100 条长工具结果消息。
- 观察打开聊天页、向上翻页、切换会话时 CPU 和 GC。
- 对比缓存前后的转换耗时。

## P2：Mermaid WebView 延迟渲染与缓存

### 现状

相关文件：

- `feature-chat/src/main/java/com/tchat/feature/chat/markdown/MermaidView.kt`
- `feature-chat/src/main/assets/mermaid.min.js`

每个 Mermaid 图表会创建 WebView，并通过 `loadDataWithBaseURL` 加载 HTML 和 `mermaid.min.js`。多个图表同屏或流式输出中图表持续变化时，开销较高。

### 建议

1. 流式期间不渲染 Mermaid。
   - 先显示代码块或简短占位。
   - 消息完成后再渲染。

2. 缓存渲染结果。
   - key：`codeHash + theme`。
   - 可缓存 SVG 字符串或截图 bitmap。

3. 限制同屏 WebView 数量。
   - 非可见或折叠状态不加载 WebView。

4. 图表默认折叠。
   - 用户点击后再渲染。

### 验证

- 单条消息包含 1、5、10 个 Mermaid 图表。
- 测试滚动流畅度、内存占用和 WebView 实例数量。

## P2：Compose 状态粒度优化

### 现状

相关文件：

- `app/src/main/java/com/tchat/wanxiaot/MainActivity.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/ChatScreen.kt`
- `feature-chat/src/main/java/com/tchat/feature/chat/MessageList.kt`

主页面收集完整 `settings`，并在同一大 Composable 内派生当前 provider、模型、工具栏设置、正则规则、知识库工具等。设置项变化时，可能触发较大范围重组。

### 建议

1. 拆分 settings Flow。
   - 当前 provider。
   - 当前模型。
   - 当前助手。
   - 语言。
   - 工具栏配置。
   - TTS/OCR/R2 等设置。

2. 使用 `distinctUntilChanged`。
   - 避免无关字段变化刷新聊天页。

3. 给聊天页建立明确 ViewModel state。
   - MainActivity 只负责路由和依赖注入。
   - ChatScreen 只接收稳定的 ChatUiState 和事件回调。

4. `LazyColumn` 使用 `contentType`。
   - 历史加载项、用户消息、助手消息、流式消息、搜索结果等区分类型。

### 验证

- 使用 Compose Layout Inspector 或 recomposition counts。
- 修改设置页中无关配置，观察聊天页重组范围。
- 流式输出时观察只有最后一条消息重组，而不是整个列表重组。

## P3：网络日志快照优化

### 现状

相关文件：

- `network/src/main/java/com/tchat/network/log/NetworkLogger.kt`
- `app/src/main/java/com/tchat/wanxiaot/ui/settings/NetworkLogScreen.kt`

`NetworkLogger.getLogs()` 每次调用都会：

```kotlin
logs.sortedByDescending { it.timestamp }
```

当前上限是 100 条，并且 Release 默认关闭，因此不是主要问题。

### 建议

如果后续提高日志上限：

- 写入时保持倒序。
- 或维护不可变快照。
- 避免日志页每次刷新都排序和分配列表。

## 建议实施顺序

### 第一阶段：低风险高收益

1. 流式期间使用纯文本渲染，完成后再 Markdown。
2. 将流式 UI 更新节流从 48ms 调整为 80-120ms。
3. `LazyColumn` 增加 `contentType`。
4. Release 开启 R8 和 `shrinkResources`，补 keep 规则。

### 第二阶段：数据规模优化

1. 消息搜索改为 FTS5 或搜索索引表。
2. 知识库 embedding 改 BLOB，并使用 topK 堆。
3. 批量查询知识库 item，消除 N+1。

### 第三阶段：结构优化

1. 消息 JSON 解析缓存或拆表。
2. Compose settings 状态拆分。
3. Mermaid 延迟渲染和结果缓存。
4. OkHttpClient 复用。

## 可接受的短期权衡

- 流式期间不完整渲染 Markdown 是合理的。用户更关注实时输出流畅，最终消息完成后再高质量渲染即可。
- 搜索先做普通索引表也可以，不必一步到位 FTS5。关键是不要在搜索时扫 JSON。
- 知识库检索可以先做 BLOB + topK 堆，再考虑近似向量索引。
- Release 开 R8 需要充分测试，但长期必须做，否则包体和冷启动会持续恶化。

3. 聊天 `LazyColumn` 增加 `contentType`，区分历史加载、普通消息、流式消息。
4. `ChatRepositoryImpl` 中聊天、消息、收藏 Flow 的实体到 UI 模型转换切到 `Dispatchers.Default`。
5. 新增 `message_search_index` 普通搜索索引表，Room schema 升级到 31；新消息写入时生成纯文本索引，旧消息在 30 -> 31 迁移中回填索引，搜索不再直接对 `messages.partsJson` 做 SQL 扫描。
6. 知识库向量检索改为 top-K 小根堆，避免全量排序，并批量查询命中的知识条目，消除 N+1 查询。
7. 构建 AI 上下文时，历史图片/视频转为文字占位，只保留当前用户消息的媒体 Base64 编码。
8. Release 构建开启 R8 和 `shrinkResources`，并补充 Room、JSON 模型、WebView JS bridge、native、Markwon/JLatexMath、ML Kit 的保守 keep 规则。

## 风险提示

- 修改消息存储结构和搜索索引需要 Room migration，并覆盖升级测试。
- R8 可能破坏反射、JSON、Markwon、ML Kit 或 native 入口，需要逐项验证。
- 知识库 embedding 从 JSON 迁移到 BLOB 时，需要处理历史数据和导入导出兼容。
- 流式 Markdown 降级要确保复制、朗读、分享、重新生成等功能仍使用完整内容。
